    Лабораторная работа №3. Проектирование БД для ИС по требованиям заказчика (PostgreSQL)

    Основа: ER-диаграмма и перечень сущностей из Лабораторной работы №2 (индивидуальный вариант на семестр)

    Формат сдачи:
    Один архив (.zip), содержащий: 1) Выгрузку БД (Backup) в одном из форматов: - .backup (custom) или - .sql 2) Короткий отчёт (PDF) с подтверждением выполнения шагов (см. раздел 2.6)

    1. Теория

    1.1. PostgreSQL и роль СУБД

    PostgreSQL — реляционная система управления базами данных (СУБД), которая хранит данные в таблицах и обеспечивает: - целостность (ограничения PK/FK/UNIQUE/NOT NULL); - согласованность связей между таблицами; - выполнение запросов SQL (SELECT/INSERT/UPDATE/DELETE); - резервное копирование и восстановление.

    В рамках ИС PostgreSQL отвечает за: - хранение справочников и документов (сущности предметной области); - поддержку бизнес-ограничений (например, уникальность номера договора); - связность данных через внешние ключи.

    1.2. Перенос ER-диаграммы в таблицы PostgreSQL

    ER-диаграмма (логическая модель) превращается в физическую схему БД по правилам:

    • Сущность → таблица
    • Customers → customers
    • Атрибут → столбец
    • email → email
    • PK → PRIMARY KEY
    • customer_id → PRIMARY KEY (customer_id)
    • FK → FOREIGN KEY
    • orders.customer_id → FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
    • 1:M
    • FK находится в таблице «M» (многие)
    • M:N
    • создаётся связующая таблица (например, order_items) с двумя FK и PK (часто составной)
    • UNIQUE
    • для бизнес-ключей: email, contract_number, sku и т.п.

    1.3. Ссылочная целостность и каскады

    Ссылочная целостность означает, что значение FK не может ссылаться на несуществующую запись.

    Типовые стратегии удаления: - ON DELETE RESTRICT / NO ACTION — запрещает удаление родителя, если есть дочерние записи; - ON DELETE CASCADE — удаляет дочерние записи вместе с родителем; - ON DELETE SET NULL — обнуляет FK в дочерних записях (если FK допускает NULL).

    Выбор стратегии должен соответствовать логике предметной области.

    1.4. Выгрузка (Backup) базы данных

    Выгрузка БД — обязательный артефакт сдачи, позволяющий восстановить вашу работу на другом компьютере.

    Варианты выгрузки: - Plain SQL (.sql) — текстовый файл с командами создания/наполнения; - Custom (.backup) — бинарная выгрузка (надёжнее для восстановления через pgAdmin).

    2. Задание

    2.1. Установка PostgreSQL на компьютер (максимально подробно)

    Шаг 1. Скачивание

    1) Откройте официальный сайт PostgreSQL (раздел Download).
    2) Выберите установщик для вашей ОС (чаще всего Windows).
    3) Скачайте установочный файл PostgreSQL.

    Шаг 2. Запуск установщика

    1) Запустите установщик от имени администратора.
    2) На шаге выбора компонентов отметьте: - PostgreSQL Server - pgAdmin 4 - Command Line Tools 3) Папку установки оставьте по умолчанию.

    Шаг 3. Пароль суперпользователя

    1) Укажите пароль для пользователя postgres.
    2) Пароль сохраните (он понадобится для входа в pgAdmin).

    Шаг 4. Порт и локаль

    1) Порт: оставьте стандартный 5432.
    2) Locale: можно оставить по умолчанию.

    Шаг 5. Завершение

    1) Дождитесь окончания установки.
    2) Откройте pgAdmin 4 (он устанавливается вместе с PostgreSQL).

    2.2. Подключение в pgAdmin 4

    1) Запустите pgAdmin 4.
    2) В левой панели найдите Servers → разверните список.
    3) Если сервер не подключён: - кликните по серверу (обычно PostgreSQL ...); - введите пароль пользователя postgres; - поставьте галочку “Save password” (по необходимости).

    2.3. Создание базы данных под ваш вариант

    1) В pgAdmin откройте: - Servers → PostgreSQL → Databases 2) Нажмите правой кнопкой на Databases → Create → Database… 3) Укажите: - Database name: is_variant_<номер> (пример: is_variant_07) - Owner: postgres 4) Нажмите Save.

    2.4. Создание схемы таблиц по ER-диаграмме (из ЛР №2)

    Вариант A (рекомендуемый): через SQL-скрипт

    1) Откройте: - ваша БД → Tools → Query Tool 2) Создайте DDL-скрипт (структуру таблиц) по вашей ER-диаграмме: - CREATE TABLE ... - PRIMARY KEY - FOREIGN KEY - UNIQUE, NOT NULL 3) Выполните скрипт кнопкой Execute (▶). 4) Проверьте, что таблицы появились: - Schemas → public → Tables

    Вариант B: ручное создание таблиц в интерфейсе

    Допускается, но требует больше времени и чаще приводит к ошибкам. Если используете ручной способ: 1) Tables → Create → Table 2) Добавляйте столбцы, затем задавайте Constraints: - Primary Key - Foreign Key - Unique

    2.5. Проверка корректности (обязательная)

    Выполните минимальные проверки:

    1) У каждой таблицы есть PK.
    2) В таблицах-связках (для M:N) есть: - два FK - составной PK или отдельный surrogate PK (по решению) 3) Все FK ссылаются на существующие таблицы и поля.
    4) В pgAdmin откройте таблицу → Properties → Constraints и убедитесь, что ограничения созданы.

    2.6. Выгрузка базы данных для сдачи (обязательно)

    Способ 1: Backup через pgAdmin (самый удобный)

    1) Правой кнопкой по вашей БД → Backup… 2) Выберите: - Format: Custom - Filename: is_variant_<номер>.backup 3) Нажмите Backup. 4) Убедитесь, что файл создался.

    Способ 2: Выгрузка в SQL (.sql) через pgAdmin

    1) Правой кнопкой по БД → Backup… 2) Format: Plain 3) Filename: is_variant_<номер>.sql 4) Нажмите Backup.

    2.7. Короткий отчёт (что включить)

    В отчёте (1–2 страницы) укажите:

    1) Название предметной области и вариант.
    2) Какая БД создана (имя базы).
    3) Список таблиц (перечень).
    4) Подтверждение ограничений: - какие PK/FK/UNIQUE применены (кратко) 5) Скриншоты: - окно pgAdmin с созданной БД и списком таблиц - окно Backup (или результат выгрузки)

    2.6. Для выполнения требуется

    1. Шаблон отчёта, указанный в описании лабораторной работы.
    2. Вариант задания, закреплённый за студентом на семестр и опубликованный в ЛМС.
    3. Требования к оформлению — согласно методическим указаниям в ЛМС или соответствующему разделу документации.
    4. Для работы требуется postress: https://www.postgresql.org/.

    2.7. Запрет

    ❗ Запрещается использование систем искусственного интеллекта для выполнения лабораторной работы.

    Работы, выполненные с использованием ИИ, полностью или частично скопированные, а также не сданные, оцениваются в 0 баллов без возможности доработки.
